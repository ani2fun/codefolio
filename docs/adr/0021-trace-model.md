# Trace model — `HeapTrace` canonicalised; harness-in-Piston for the JVM tracer

`HeapTrace` (ADR-0018, shipped for Python) is promoted to the *single* trace model for every language we visualise. Three parallel shapes — `HeapTrace`, the `Snapshot` from `feature_visualise.md`, and the `TraceStep` from `IMPLEMENTATION_PROMPT.md` — drifted across design docs without anyone shipping a non-`HeapTrace` consumer; this ADR retires the latter two. `HeapStep.fn` + `HeapStep.locals` collapse into `HeapStep.frames: List[HeapFrame]`, head-as-active, leaving room for recursion-aware UI without a further schema change. `ArrKind` gains `JArr` so native Java arrays are a third sequence kind beside `Lst` / `Tup` (the renderer's badge / bracket styling is all that meaningfully differs between the three).

The execution mechanism for JVM tracing is **harness-in-Piston**: client-side `JvmTracer.wrap` emits a self-contained Java program that uses `javax.tools.JavaCompiler` + `com.sun.source.util.TaskListener` to rewrite the user's source, runs it in the existing Piston / Code Runner sandbox, and prints marker-delimited JSON. This mirrors the Python `sys.settrace` harness pattern; there is no server-side tracer service, no JDWP/JDI, and no second JVM fork. Slice 0's capability probe confirmed both backends expose `jdk.compiler` and `jdk.attach`, so the mechanism is unblocked. Java is the only traced JVM language in v1 — Kotlin and Scala remain source-display tabs with a `LanguageLockBanner` (paralleling `IMPLEMENTATION_PROMPT.md` §6.6 but with no tracer behind them).

## Considered alternatives (and why rejected)

- **Three parallel trace shapes, one per language.** Rejected — diverges every downstream consumer (the D3 renderer's adapter, the locals panel, the modal's code-pane sync). The shapes had already drifted across docs without anyone shipping a non-`HeapTrace` consumer; this codifies the convergence rather than fragmenting on speculative future needs.
- **Server-side JDWP/JDI tracer** (the `feature_visualise.md` plan). Rejected as the v1 mechanism — it needs a long-running Scala service forking a child JVM with `dt_socket` per request, a transport adapter, lifecycle / sandboxing management, and a queue. The harness-in-Piston pattern ships server-side-change-free; the existing `/api/run` already routes Java through Piston / Code Runner. The server-side path remains a viable v2 if Piston's sandbox eventually rejects `jdk.compiler` or its stdout cap becomes the bottleneck — but until that constraint binds, "more code, more failure modes" is not a justified migration.
- **In-process JDI via `LaunchingConnector` inside the Piston sandbox.** Rejected — Piston sandboxes routinely disallow second-JVM forks with `dt_socket` bindings. The probe confirmed `com.sun.tools.attach.VirtualMachine` is reachable in the current Code Runner image, but the surface a self-attached debugger needs is larger and more failure-prone than the compiler-AST surface.
- **Embedded ASM + `jdk.attach` self-attach as the primary mechanism.** Rejected as the v1 default — the ASM library would ship as a ~250 KB base64 blob in every `JvmTracer` payload, and bytecode-level traversal is harder to reason about than `com.sun.source.tree`. Kept as the **declared fallback** if a future Piston image strips `jdk.compiler` from the runtime: the schema and downstream UI don't care which mechanism produces the JSON, so the fallback substitution is purely a `wrap()` change.
- **`HeapStep.fn` + `HeapStep.locals` left at the top level + a separate stack field.** Rejected — two sources of truth for "what's the current frame?" would force every consumer to know which to prefer. Collapsing into `frames.head` makes the active-frame-only behaviour read as `step.frames.headOption.fold(Nil)(_.locals)` everywhere, and the recursion-aware UI becomes a renderer addition that doesn't break callers.
- **Add a `JArr` variant of `HeapObject` rather than extending `ArrKind`.** Rejected — `HeapObject.Arr` already carries the "kind tag + items" shape; the only meaningful difference between a Python list, a Python tuple, and a Java array is the badge / brackets the renderer prints around the cells. A new top-level variant would force adapter branches in `HeapToGraph.nodesOf` / `edgesOf` / `outRefs` for no semantic gain.

## Consequences worth calling out

- **Active-frame-only behaviour is unchanged today.** Every downstream consumer (`HeapToGraph`, `TracedCodeBlock.renderLocalsPanel`, the future `JvmTracer` decoder) reads `step.frames.headOption.fold(Nil)(_.locals)` and `step.frames.headOption.fold("")(_.fn)`. No visible behaviour changes when this ADR lands — the migration is structural. A recursion-aware UI (a stack panel, a breadcrumb of pending frames) becomes a follow-up that adds renderer code without touching the schema.
- **Python harness walks `frame.f_back`.** `PythonTracer.wrap` is updated in the same slice to populate `frames` by walking the frame chain, filtering to traced-file frames, innermost first. The JVM harness will have the stack natively from the AST walker's method-tree stack, so both sides emit a populated `frames` list from the outset.
- **`HeapToGraphSpec` migrates structurally.** All fixtures that build `HeapStep(line, "line", "fn", locals, heap)` switch to `HeapStep(line, "line", List(HeapFrame("fn", locals)), heap)`. Behavioural assertions are unchanged; `isHelperFrame` consults the innermost frame's name (the function whose code is being stepped through) — outer caller frames don't decide whether the step is dropped.
- **`HeapToGraph.HelperFnPrefixes`** stays Python-flavoured today (`from_`, `to_`, `build_`, `__init__`). Slice 5 extends it with `<init>` and common Java conventions (`from*`, `build*`, etc.) without a further ADR — the prefix list is implementation detail, not a contract.
- **`/api/run` is untouched.** The JVM harness is a `.java` source string; the server endpoint, the wire adapters, and `Languages.effectiveSource` see it as any other Java submission. The only server-side addition the broader plan eventually carries is the Slice 2 tracer sentinel (`// __CF_TRACER__`) so wrapped sources skip `JavaSourceRewriter.normalizeEntrypoint`.
- **`BackendLimits` arrives in Slice 2, not here.** The harness needs to size its `_cf_max_payload`-equivalent against the active backend's stdout cap; that table is its own small surface and ships parallel to this schema work.
- **Legacy spec docs get HISTORICAL banners** rather than deletion. `docs/new_feature/feature_visualise.md` and `docs/new_feature/claude_design/handoff_bundle/visualise/IMPLEMENTATION_PROMPT.md` keep their authored content (the UX shape, modal chrome, keyboard shortcuts, multi-case selector are still load-bearing references for later slices) but acquire a leading banner pointing at this ADR for the resolution of the `Snapshot` / `TraceStep` / `HeapTrace` contradiction and the JDWP-vs-harness mechanism reversal. Both files are local-only (gitignored); the banner is for future-me reading the working tree.

## Deferred to a later ADR

- **Recursion-stack UI** — a stack panel showing pending frames, a breadcrumb above the locals table, click-to-pin a frame. Mechanically this reads `step.frames` directly; no schema change. Scope-out until at least one DSA chapter ships a recursive `java trace` snippet whose pedagogy the current single-frame panel underserves.
- **Kotlin / Scala native tracers.** Mechanism choice is open — likely ASM-based bytecode rewriting since both compile to JVM but neither exposes a stable source-AST surface comparable to `com.sun.source.tree`. Until a chapter genuinely needs a Kotlin / Scala trace, source-only tabs with `LanguageLockBanner` are the v1 answer.
- **Server-side traced-run scale-up.** A queue-based worker pool (self-hosted Code Runner / Judge0 at scale) is its own plan. The harness pattern remains correct under either backend; the work is operational, not architectural, and is tracked as Slice 8 (parallel, optional) in the JVM-tracer plan rather than as an ADR.
