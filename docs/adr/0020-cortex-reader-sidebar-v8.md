# Cortex reader sidebar — V8 (slide-to-resize) replaces V7 rail + mini-map

The pre-V8 reader had two parallel chrome surfaces — a left sidebar with a binary collapse / hover-peek rail, and a right-edge mini-map (`CortexMiniMap`). V8 collapses both navigation surfaces into one: a continuously resizable left sidebar (`[160, 520]` px, persisted to `localStorage["cortex.sidebarWidth"]`, rail mode under 200 px), with per-chapter read-state dots, an `::after` progress bar on the active row, and a sticky breadcrumb that fades in only when the active row scrolls out of view. The mini-map is deleted, not replaced — the implicit claim is that the sidebar dots + active progress bar + preserved right-side TOC popover (`.cortex-reader-toc-pop`) collectively subsume it, and the only thing genuinely lost is the mini-map's spatial-recall-of-scroll-position affordance (an acceptable trade in exchange for not maintaining two parallel navigation chromes).

## Considered alternatives (and why rejected)

- **Keep the binary collapsed-toggle, add no mini-map.** Rejected — the prose column position is a dominant visual concern in a reading app, and a 3-state cycle (rail / default / wide) forces a wrong-for-everyone width on most readers. A continuum lets each reader settle on their own sweet spot. Justifies the ~100+ LOC + ARIA `role="separator"` + persistence cost because this is a *reading* surface used for hours, not a navigation utility.
- **Keep the mini-map alongside V8.** Rejected — two parallel navigation chromes means two places to maintain the active-state highlight, two scroll listeners, and visual competition for the reader's eye. The mini-map's *unique* affordance (spatial-recall of "where in this chapter did I see X?") is an edge case; the right-side TOC popover handles the common case of "jump to a known section".

## Consequences worth calling out

- **Read Progress** is single-device by design (CONTEXT.md term). localStorage-only, no server sync, even though Keycloak auth is in place. A future "synced library" would be a separate feature, not a transparent upgrade.
- **Active-ancestor highlighting safety** rests on chapter-slug uniqueness propagating up through `SidebarForest.Section` subtree-uniqueness. Two structurally-equal Sections would compare `==` under Scala case-class structural equality, but in practice can't, because their subtrees include unique chapter slugs. Documented inline at `SidebarForest.Section`.
- **Live `--p` progress bar** on the active sidebar row is fed by an rAF-coalesced (not debounced) scroll-fraction signal; the *persisted* `cortex:progress:<slug>` write keeps its 200 ms debounce. Two signals, deliberately split — the live bar tracks the eye, the persisted value records the high-water mark.
- **Legacy `cortex-reader-sidebar__*` CSS namespace coexists with `sb8__*`** for the desktop wrapper / mobile drawer chrome. The V8 spec said "replace, don't keep parallel namespaces"; the in-flight code obeys that for sidebar-body internals but keeps the outer-chrome classnames. Acceptable because the outer wrapper is mode-switching (drawer vs. desktop), not a duplicate of any V8 component.
