# Cortex diagrams pick a format by purpose — D3 for dynamic state, D2 for static structure, Mermaid for algorithm flow

Cortex chapters today reach for whichever of `mermaid`, `d2`, or `d3 widget=…` feels right per author per session, with no rule about which fits which job. The result, surfaced during the singly-linked-list audit (chapters in [content/cortex/data-structures-and-algorithms/02-linear-structures/03-singly-linked-list/](../../content/cortex/data-structures-and-algorithms/02-linear-structures/03-singly-linked-list/)): adjacent diagrams of the *same* concept render in different formats, the reader has to re-learn what a node looks like at each page break, and the same algorithm shows up once as a stepped D3 widget and once as a static D2 box for no defensible reason. ADR-0006 settled the *catalog* of D3 widgets; ADR-0013/0014 settled how they animate. Nothing said when to reach for D3 versus D2 versus Mermaid in the first place.

**Decision**: each Block format has one job. Authors pick by what the diagram is *showing*, not by which they happened to use last.

| Showing | Use |
|---|---|
| Data-structure **state over time** — insert, delete, reverse, two-pointer walk, anything with a sequence of frames where nodes/cells/pointers move | **D3 widget** from the closed catalog (`linked-list`, `array-traversal`, `btree-walker`, …) |
| **Static structural figure** — node anatomy (`val + next` fields), memory layout, single annotated snapshot of a list/tree, a system diagram | **D2** static block (` ```d2 `) |
| **Algorithm control flow** — `init → loop → end`, decision branches, recursion outline, pseudocode-as-a-graph | **Mermaid** flowchart (` ```mermaid `) |

  - **Why split these three, not unify on D3.** A one-step D3 widget for a static figure is silly: it ships a Scala component, a payload schema, and a step-controls strip the reader can't use, all to render what a 12-line d2 block draws server-side at build. D2's static output is also higher fidelity for structural figures — the WASM pre-render gives crisp text and arbitrary shapes that a hand-rolled SVG generator inside a widget would have to re-invent. Mermaid's flowchart vocabulary (`flowchart LR`, `[…]`, `(…)`, `{…}`) is purpose-built for the *while-loop-with-if-branches* shape that recurs every time a chapter explains an iterative algorithm; D2 can draw flowcharts too but the syntax is heavier and the layout less reliable for the branching shape.
  - **Why the rule is *strict by purpose*, not *whatever the author prefers*.** The drift this fixes is visible at the section level, not the chapter level. A reader scrolling through 13 linked-list chapters sees the same concept (a 3-node list) drawn three ways depending on which chapter they're on; that's the inconsistency the author can't see from inside one chapter. Locking the format to the concept removes the choice — the chapter's job is to show the concept; the *format* falls out of what's being shown.
  - **A chapter may use all three**, but never two formats for the *same* content. The full-list reversal chapter, for instance, uses a D3 `linked-list` widget for the three-pointer walk (state over time), a D2 block for the node-anatomy figure that opens the section (static structure), and a Mermaid flowchart for the `while current is not null: …` loop (control flow). Each format does the job it was picked for.
  - **No author-side theming inside Mermaid blocks.** The runtime theme initialiser in `MermaidBlock` already resolves the light/dark palette from the page's `<html class>`; chapters that ship inline `themeVariables` config blocks (the pattern in some current chapters) override the global theme and produce inconsistent colours across the section. Drop them; let the runtime pick.
  - **Where this lives**: codified in [docs/authoring-cortex-blocks.md](../authoring-cortex-blocks.md). The widget catalog in [`D3WidgetBlock.scala`](../../client/src/main/scala/codefolio/client/components/cortex/D3WidgetBlock.scala) and the [`Block`](../../shared/src/main/scala/codefolio/shared/cortex/Blocks.scala) ADT stay unchanged — this is a *content* rule that applies above the rendering layer.

The trade is **a rule the author has to remember** for **a section-wide visual coherence the reader gets for free** — and a clear answer when a future widget review asks "why is this concept rendered in two formats?".
