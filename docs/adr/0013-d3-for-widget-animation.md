# D3 powers widget animation; widgets render via React-managed refs, not SVG strings

ADR-0006 introduced widgets as a closed catalog of "Scala.js + D3 components" embedded by name from markdown. In practice the first round of widgets — `array-traversal`, `latency-scaled-time`, `partition-simulator`, and the rest — were built without an actual D3 dependency: each step rebuilds its SVG as a string and re-injects it via `dangerouslySetInnerHtml`. That works for static visuals where the only per-step change is which `<rect>` gets a highlight class, but it cannot animate values that *move* between cells. Two-pointer problems (Flip Characters, Vowel Exchange, Reverse Segments, …) are exactly that case: the items themselves swap positions, and a string rebuild produces a hard jump rather than the visual continuity that makes the swap legible.

**Decision**: bring D3 in for real, scoped to widget animation. Widgets that need cell-level continuity move from "rebuild the SVG string each step" to "let D3 own the SVG subtree and run transitions on selection updates."

  - **Library choice — D3 over Visx, Plot, Konva, P5.** D3's selection-and-transition model is exactly the shape we need: data-bound `<rect>`/`<text>` nodes that *persist across step changes* and *animate their attribute deltas*. Visx wraps D3 in React components — useful when authoring in TSX, but actively annoying from Scala.js where we'd be writing facades around a wrapper. Plot is high-level and chart-centric; widgets aren't charts. Konva is canvas, which kills selectable text and inflates SVG-bound test mental models. P5 is a creative-coding framework, not a visualisation toolkit. D3 is also already implied by ADR-0006's name; bringing it in finally closes a long-standing gap between the ADR's intent and the code.
  - **Animation pattern — D3 selections inside React effects.** Each animating widget keeps a `useRef` to its SVG element. A `useEffectWithDepsBy` on `(step, payload)` runs the D3 update: `d3.select(svgRef).selectAll(".cell").data(items, keyFn).join(enter, update, exit).transition().duration(N).ease(easeCubicInOut).attr("x", ...)`. React owns the *frame*; D3 owns the *contents*. Step controls (prev/next/play/reset) stay React state. This replaces the current pattern's full rebuild on every step.
  - **Packaging — full `d3` to start, budget-gated migration to submodules.** `npm install d3` adds ~70 KB gzipped to the client bundle (the JS payload — no `@types/d3` because we're not consuming D3 from TypeScript). The cost matters but it's not actionable until we're closer to the budget ceiling: this ADR sets a soft cap of **+90 KB gzipped** above the pre-D3 baseline. When we cross the cap, migrate to selective submodules (`d3-selection`, `d3-transition`, `d3-ease`, and whatever the catalog has accreted by then). Don't pre-optimise — half the catalog hasn't been refactored yet, so we don't know which submodules we'll need.
  - **Scala.js facade — grow on demand.** A new `codefolio.client.d3` package houses `@js.native` + `@JSImport` bindings. Initial scope: `select`, `selectAll`, `Selection`, `Transition`, `easeCubicInOut`. New widgets extend the facade only when they need new D3 surface area. Refuse the temptation to write a complete typed facade up front; D3's API is large and most of it stays unused.
  - **Backward compatibility.** Widgets that don't need animation continue to use the string-rebuild pattern as long as it works for them. The D3 path is opt-in per widget. ADR-0006's closed-catalog rule stands: rendering mechanism is a widget's private business.

The trade is **a meaningful client dep** (~70 KB gzipped, a typed facade to maintain, a second rendering pattern in the catalog) for **visual continuity on swap-style widgets** — which the two-pointer chapter cannot get from string rebuilds, and which the rest of the algorithms book is going to keep demanding (sliding windows, partitioning, merging). The closed-catalog discipline from ADR-0006 absorbs the second pattern without leaking: authors still just name a widget and pass JSON; the catalog renderer decides whether D3 or SVG strings runs underneath.
